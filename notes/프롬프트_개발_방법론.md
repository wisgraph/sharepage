# 프롬프트 개발 방법론

## 불완전한 실행으로부터 수렴하는 설계의 기술

---

## 서문

이 문서는 프롬프트를 **한 번에 잘 쓰는 법**에 관한 것이 아니다. 그런 방법은 없다. 대신 **불완전한 프롬프트를 점진적으로 다듬어 원하는 결과물에 도달하는 체계**에 관한 것이다.

많은 사람이 프롬프트 작성을 "AI에게 작업을 명확히 지시하는 글쓰기"로 이해한다. 이 관점은 절반만 맞다. 프롬프트는 글쓰기인 동시에 **설계**이고, 설계는 한 번에 완성되지 않는다. 건축가가 머릿속에서 완벽한 건물을 그린 뒤 도면 한 장에 담는 게 아니라, 스케치와 모형과 수정을 거치며 설계를 수렴시키듯, 프롬프트도 같은 과정을 거친다.

이 문서의 출발점은 하나의 문장이다.

> **프롬프트는 한 번에 완성되는 게 아니라, 불완전한 실행을 통해 드러나는 문제들을 체계적으로 해소해가며 수렴하는 설계물이다.**

여기에는 세 가지 핵심 키워드가 있다.

- **"불완전한 실행"** — 완벽하지 않아도 일단 돌려봐야 한다
- **"드러나는 문제"** — 문제는 사전에 다 예측 못 하고, 실행 후에 보인다
- **"체계적 해소"** — 감으로 수정하는 게 아니라 검증된 도구와 절차를 쓴다

이 세 가지를 리듬으로 가져갈 수 있으면, 어떤 종류의 프롬프트든 개발할 수 있다. 학습용이든, 글쓰기 어시스턴트든, 분석 도구든, 창작 파트너든.

---

## 이 문서의 구성

문서는 4개 층위로 구성된다. 각 층은 추상에서 구체로 내려간다.

1. **근본 원리** — 왜 그렇게 해야 하는가
2. **사고 도구** — 어떻게 생각해야 하는가
3. **실전 워크플로우** — 언제 무엇을 할 것인가
4. **함정과 대처** — 무엇을 피할 것인가

순서대로 읽는 것을 권장하지만, 이미 경험이 있다면 3층(워크플로우)부터 보고 필요할 때 1~2층을 참조해도 좋다. 4층은 언제든 문제가 생겼을 때 펼쳐보는 트러블슈팅 가이드로 활용할 수 있다.

# 1층: 근본 원리

모든 방법론의 뿌리다. 이 원리들을 이해하면 개별 기법은 자연스럽게 도출된다. 거꾸로 원리 없이 기법만 따라하면 상황이 바뀔 때 응용이 안 된다.

## 원리 1: 프롬프트는 "명세"가 아니라 "인터페이스"다

### 통상적 이해

대부분의 사람은 프롬프트를 **AI에게 주는 작업 명세서**로 생각한다. 이 관점에서 좋은 프롬프트는 "작업을 빠짐없이 자세히 적은 문서"다. 그래서 프롬프트가 길어지고, 조항이 늘어나고, 예외 처리가 더해진다.

이 이해는 틀린 건 아니지만 불완전하다.

### 더 정확한 모델

프롬프트는 **사용자와 AI 사이의 인터페이스**다. 인터페이스는 명세와 달리 **양방향 상호작용을 설계**하는 것이다. 세 가지 차원을 동시에 고려해야 한다.

- **입력 설계**: 사용자가 이 프롬프트를 어떻게 호출하는가? 어떤 형태의 입력을 주게 되는가?
- **처리 설계**: AI가 그 입력을 받아 어떻게 해석하고 어떻게 생성하는가?
- **출력 설계**: 결과물이 사용자에게 어떤 형태로 도착하고, 사용자는 그것을 어떻게 소비하는가?

### 실제 사례: "의도 파악" 섹션의 재해석

한 학습 프롬프트 개발 과정에서 초기 버전은 "AI가 항상 먼저 질문한다"는 규칙을 포함했다. 사용자가 맥락을 충분히 제공해도 AI는 매번 "어떤 맥락에서 접하셨나요?"부터 물었고, 이것이 불편함을 만들었다.

- **명세 관점의 진단**: "질문 단계의 로직이 잘못 쓰였다"
- **인터페이스 관점의 진단**: "사용자 입력을 제대로 활용하지 못하는 인터페이스 결함"

두 진단의 해결책은 다르다. 전자는 "질문 문구를 수정하자"로 가기 쉽고, 후자는 "사용자 입력을 먼저 파싱해서 이미 주어진 정보는 재확인하지 않도록 구조 자체를 바꾸자"로 간다. 후자가 본질적 해결이다.

### 실천적 함의

프롬프트를 쓸 때 다음 세 가지를 의식적으로 상상하라.

1. **호출 시점**: 사용자가 이걸 어떻게 쓰게 되는가? 매번 긴 맥락을 쓸까, 짧게 던질까?
2. **반응 경험**: AI의 출력이 사용자에게 어떻게 느껴질까? 도움이 될까, 번거로울까?
3. **마찰 지점**: 어디에서 사용자가 "아, 이거 귀찮네"라고 느낄까?

인터페이스 디자이너처럼 생각하라. 명령문을 쓰는 게 아니라 상호작용을 설계하는 것이다.

## 원리 2: LLM은 "지시를 따르는 기계"가 아니라 "맥락에 반응하는 기계"다

### 통상적 이해

"AI에게 명확히 지시하면 그대로 수행한다"는 모델이다. 이 관점에서 좋은 프롬프트는 **명령문의 집합**이다. "이렇게 해라, 저렇게 해라, 이건 하지 마라."

이것도 틀린 건 아니다. 명령문은 작동한다. 하지만 최적은 아니다.

### 더 정확한 모델

LLM은 근본적으로 **맥락 속에서 가장 그럴듯한 다음 말을 생성하는 시스템**이다. 명령을 실행하는 게 아니라 맥락에 반응한다. 이 차이가 실제로 의미하는 바는 크다.

**맥락 설정이 명령 나열보다 강력하다.** 예를 들어:

- **명령형**: "친절하게 답변해라. 공감을 표현해라. 재촉하지 마라."
- **맥락형**: "너는 20년 경력의 학습 코치이고, 조급한 학습자를 안정시키는 게 전문이다."

두 프롬프트 모두 비슷한 방향을 지향하지만, 맥락형이 훨씬 일관된 결과를 낸다. 이유는 단순하다. 맥락형은 **AI가 자연스럽게 그 방향으로 생성하게 될 상황**을 만들어주고, 명령형은 **기본 성향에 제약을 덧씌운다**. 전자는 물의 흐름을 이용하는 것이고, 후자는 물을 억지로 위로 올리는 것이다.

### 왜 이것이 실제로 효과가 있는가

LLM의 훈련 데이터에는 수많은 "역할"과 "상황"과 "전문가"의 발화가 들어 있다. "20년 경력 학습 코치"라는 맥락을 받으면, 모델은 그런 사람이 쓸 법한 어휘, 문장 구조, 태도, 세심함을 자동으로 활성화한다. 별도의 명령 없이도 일관되게.

반면 명령문은 매번 "이걸 지켜야 해"라는 부하를 모델에 주는 것과 비슷하다. 긴 대화에서 명령의 힘은 약해지기 쉽다.

### 실천적 함의

**문제가 생겼을 때의 첫 반응**이 바뀌어야 한다.

- **초보자**: "지시를 더 추가하자" → 금지 조항 10개 나열
- **숙련자**: "맥락을 다시 설계하자" → 상황·역할·배경 재정의

금지 조항 10개보다 "이런 상황에서 이런 태도로 일하는 사람"이라는 맥락 한 줄이 더 강하다. 항상 그런 건 아니지만, 원칙으로 기억할 가치가 있다.

## 원리 3: 모든 프롬프트에는 "암묵적 가정"이 숨어 있다

### 통상적 이해

프롬프트를 쓸 때 우리는 "필요한 건 다 적었다"고 느낀다. 하지만 실제로는 수많은 것을 **당연하게 전제**하고 있다. 사용자가 어떤 사람일지, 어떤 맥락에서 쓸지, 어떤 결과를 원할지에 대한 가정들이 명시되지 않은 채 프롬프트에 반영된다.

### 암묵적 가정의 특징

가정은 **문제가 생기기 전까지는 보이지 않는다.** 이게 핵심이다.

- "사용자는 진지하게 학습하고 싶어한다" → 대화형 구조가 당연해 보임
- "사용자는 충분한 시간이 있다" → 긴 응답이 문제없어 보임
- "사용자는 해당 분야에 관심이 있다" → 기본 설명을 생략함
- "사용자는 한국어 화자이고 한국 맥락에서 쓴다" → 지역 변수를 고려 안 함

각 가정은 그 자체로 합리적일 수 있다. 하지만 **모든 사용자에게 참은 아니다**. 그리고 그 사실은 실제로 다른 유형의 사용자가 써보기 전까지는 드러나지 않는다.

### 가정 발굴의 기법

가정은 저절로 보이지 않는다. 의도적으로 들춰내야 한다. 두 가지 질문이 유용하다.

**질문 1: "누가 이걸 쓸 것인가?"**

한 사람만 상상하지 말고, 최소 세 명의 다른 사용자를 상상하라. 이 셋이 같은 프롬프트를 썼을 때 모두 만족하는가? 누군가에게는 너무 길고, 누군가에게는 너무 얕고, 누군가에게는 맥락이 안 맞지 않는가?

**질문 2: "내가 당연하게 여긴 것은 무엇인가?"**

프롬프트를 쓰고 나서 의도적으로 질문하라. "여기서 내가 뭘 전제했지?" 짧게라도 가정 목록을 적어보면, 그중 일부가 취약하다는 것을 발견한다.

### 실천적 함의

암묵적 가정을 완전히 제거할 수는 없다. 모든 것을 명시하면 프롬프트가 무한히 길어진다. 대신 **중요한 가정은 의식적으로 선택**해야 한다.

- "이 가정을 유지할 것인가, 유연하게 만들 것인가?"
- "유연하게 만든다면, 어떻게 분기 처리할 것인가?"
- "유지한다면, 그 가정이 깨지는 사용자는 어떻게 안내할 것인가?"

가정을 **의식화**하는 것 자체가 이미 큰 진전이다.

## 원리 4: 좋은 프롬프트는 "결정"이 아니라 "결정의 집합체"다

### 통상적 이해

프롬프트를 **하나의 덩어리**로 본다. 좋은 프롬프트 vs 나쁜 프롬프트, 긴 프롬프트 vs 짧은 프롬프트, 자세한 프롬프트 vs 간결한 프롬프트. 이런 이분법으로 생각한다.

### 더 정확한 모델

한 편의 프롬프트 안에는 **수많은 독립적 결정**이 들어 있다.

| 결정 차원     | 선택지 예시                      |
| ------------- | -------------------------------- |
| 역할          | 전문가 / 코치 / 친구 / 중립적 AI |
| 톤            | 격식 / 친근 / 분석적 / 창의적    |
| 길이          | 초간결 / 적정 / 상세 / 장문      |
| 구조          | 자유 / 고정 템플릿 / 동적 구조   |
| 상호작용      | 대화형 / 원샷형 / 단계형         |
| 예시          | 없음 / 1~2개 / 다수              |
| 에러 처리     | 엄격 / 유연 / 무시               |
| 출력 형식     | 텍스트 / 마크다운 / JSON / 혼합  |
| 불확실성 처리 | 명시 / 숨김 / 근사               |

숙련된 프롬프트 작성자는 이것들을 **독립적으로 조정 가능한 변수**로 인식한다. 하나를 바꾼다고 다른 것이 따라 바뀌지 않는다.

### 모듈화의 실질적 이점

**이점 1: 개선 범위가 명확해진다.** "깊이를 줄이자"는 요구가 왔을 때, 덩어리로 인식하면 어디를 건드려야 할지 모호하다. 결정 집합으로 인식하면 "길이 차원에서 '상세'를 '적정'으로 낮춘다"로 구체화된다.

**이점 2: 부작용을 예측할 수 있다.** 한 결정을 바꾸면 어떤 연쇄 효과가 올지 예상 가능하다. "상호작용을 대화형에서 원샷형으로 바꾸면 길이가 자연히 늘어나고, 중간 확인 루프가 사라지므로 에러 처리 방식도 재검토해야 한다"는 식으로.

**이점 3: 재사용이 가능해진다.** 잘 분리된 결정들은 다른 프롬프트에도 가져다 쓸 수 있다. "원샷형 + 동적 구조" 조합이 학습 프롬프트에서 잘 작동했다면, 분석 프롬프트에도 시도해볼 수 있다.

### 실천적 함의

프롬프트를 쓸 때 **어떤 결정들이 이 안에 들어있는가**를 명시적으로 의식하라. 가능하면 섹션이나 주석으로 분리하라.

```
# 역할 정의
[역할과 관련된 결정들]

# 상호작용 방식
[대화/원샷/단계 관련 결정들]

# 출력 구조
[형식과 길이 관련 결정들]

# 예외 처리
[에러와 경계 케이스 관련 결정들]
```

이런 구조화가 있으면 **"이것만 바꾸고 저건 그대로"**가 가능해지고, 수정이 국소적으로 이루어진다. 전체를 다시 쓰지 않아도 된다.

# 2층: 사고 도구

원리를 알았다고 해서 바로 좋은 프롬프트가 나오지는 않는다. 원리와 실전 사이에는 **사고 도구**가 필요하다. 도구는 원리를 구체적인 사고 절차로 변환한다.

아래 여섯 개 도구는 서로 독립적이지만, 상황에 따라 조합해서 쓰는 것이 효과적이다. 각 도구는 특정 원리에 뿌리를 두지만, 실전에서는 원리를 의식하지 않고도 도구를 쓸 수 있다.

## 📊 사고 도구의 지도

```merlight
graph LR
    subgraph Users["사용자 이해"]
        T1[페르소나 다중화]
        T4[최소 호출 원칙]
    end

    subgraph Design["설계 검증"]
        T2[반전 테스트]
        T3[실패 시나리오]
    end

    subgraph Output["출력 설계"]
        T5[레이어드 아웃풋]
        T6[제약을 자유로]
    end

    Users --> Design
    Design --> Output
    Output -.피드백.-> Users

    style Users fill:#fff3cd
    style Design fill:#d1ecf1
    style Output fill:#d4edda
```

각 도구는 프롬프트 개발의 서로 다른 단계에서 빛난다. 사용자를 이해하는 단계, 설계를 검증하는 단계, 출력을 다듬는 단계에서 각각 다른 도구를 꺼내 쓰게 된다.

---

## 도구 1: 사용자 페르소나 다중화

### 언제 쓰는가

프롬프트의 초기 설계 단계, 그리고 "누구에게 어떤 경험을 줄 것인가"를 결정할 때.

### 핵심 아이디어

**한 명의 사용자**만 상상하며 프롬프트를 쓰지 마라. 최소 세 명의 다른 사용자를 상상하라.

- **최소 사용자**: "딱 필요한 만큼만 빨리"를 원하는 사람
- **중간 사용자**: 적정 깊이로 이해하고 싶은 사람
- **최대 사용자**: 끝까지 파고들고 싶은 사람

이 셋이 같은 프롬프트를 썼을 때의 경험을 시뮬레이션한다.

### 실전 적용

학습용 프롬프트를 만든다고 가정하자. 한 명만 상상하면 보통 "진지하게 학습하는 사람"을 떠올리고, 그에 맞춰 장문의 구조적 응답을 설계한다. 하지만 세 명을 상상하면 다른 그림이 보인다.

- **최소 사용자**: "블록체인? 뭔지만 대충 알면 돼" → 장문 응답은 부담
- **중간 사용자**: "블록체인 제대로 이해하고 싶다" → 장문 응답 환영
- **최대 사용자**: "블록체인 실무에 적용해야 한다" → 장문 응답이 오히려 부족

해결책은 세 가지 중 하나다.

1. **특정 사용자에 최적화** (다른 두 사용자는 포기)
2. **공통분모로 설계** (모두에게 적당하지만 누구에게도 최적은 아님)
3. **분기 처리** (목표 수준에 따라 다른 응답)

세 번째가 가장 좋지만 구현 복잡도가 가장 높다. 선택은 프롬프트의 용도에 따라 다르다.

### 주의점

페르소나를 지나치게 세분화하면 오히려 설계가 마비된다. **세 명으로 시작하고, 필요할 때만 더 추가**하라. 실제로는 세 명의 페르소나만 잘 상상해도 대부분의 결함은 발견된다.

---

## 도구 2: 반전 테스트

### 언제 쓰는가

프롬프트의 근본 성격을 의심해야 할 때. 세부 조정으로 해결 안 되는 불편함이 지속될 때.

### 핵심 아이디어

프롬프트의 **핵심 결정을 반대로 뒤집어본다**.

- 대화형을 선택했다면? → 원샷으로 뒤집어보자
- 상세를 선택했다면? → 간결로 뒤집어보자
- 범용을 선택했다면? → 특화로 뒤집어보자
- 고정 구조를 선택했다면? → 동적 구조로 뒤집어보자

반전시킨 버전을 실제로 써보거나 최소한 머릿속으로 돌려본다.

### 결과의 두 가지 가능성

**가능성 1: 반전이 더 낫다** → 기존 설계의 근본 성격을 바꿀 때가 왔다는 신호. 세부 수정으로는 도달할 수 없는 도약이 가능하다.

**가능성 2: 원본이 여전히 낫다** → 그 이유가 명확해지면, 그것이 설계의 **근거**가 된다. 이전엔 "왠지 이게 맞아 보여서" 선택했던 것이, 이제는 "반전보다 이게 나은 이유가 X, Y, Z"로 정당화된다.

둘 다 가치 있는 결과다. 반전 테스트에 "실패"는 없다.

### 실전 적용

한 프롬프트를 개선하던 중 "의도 파악 단계가 불편하다"는 피드백이 반복됐다. 세부 수정으로는 만족이 안 됐다. 반전 테스트를 시도했다.

- **원본**: "AI가 먼저 질문하고 대화로 진행한다" (대화형)
- **반전**: "AI가 입력에서 모든 걸 추론하고 한 번에 결과를 낸다" (원샷형)

반전 버전이 훨씬 나았다. 세부 조정으로는 절대 도달 못 할 전환이었다. 이후 프롬프트의 전체 구조가 이 전환에 맞춰 재설계됐다.

### 주의점

반전 테스트는 **리소스가 든다.** 매번 하면 피로해진다. 대신 아래 신호가 올 때 꺼내 쓰라.

- 비슷한 수정을 3번 이상 반복했는데도 만족스럽지 않을 때
- 사용자 피드백이 "구조 자체"를 건드리는 것일 때
- 새로운 사용 방식이나 맥락이 나타났을 때

---

## 도구 3: 실패 시나리오 생성

### 언제 쓰는가

프로토타입을 만든 직후, 그리고 실전에 투입하기 전.

### 핵심 아이디어

"이 프롬프트가 망하는 경우"를 **적극적으로 상상**한다. 방어적이 아니라 공격적으로. AI가 최대한 나쁘게 반응할 수 있는 조건을 생각해낸다.

### 점검할 시나리오 유형

**유형 1: 비정상 입력**

- 사용자가 프롬프트가 예상한 것과 전혀 다른 입력을 하면?
- 입력이 너무 짧거나, 너무 길거나, 다른 언어이거나, 무의미하면?
- 입력에 오타나 모호함이 있으면?

**유형 2: 해석의 분기**

- AI가 프롬프트를 내 의도와 다른 방향으로 해석할 가능성이 있는가?
- 하나의 문장이 두 가지 의미로 읽힐 수 있는가?
- "자세히"나 "간략히" 같은 주관적 표현이 있는가?

**유형 3: 길이 및 일관성**

- 출력이 길어지다가 중간에 품질이 떨어지지 않는가?
- 초반과 후반의 톤이 일관되는가?
- 장문 응답에서 초반 지시를 잊지 않는가?

**유형 4: 편차**

- 같은 프롬프트로 10번 실행했을 때 결과가 얼마나 다른가?
- 편차가 허용 범위인가?

### 실전 적용

각 실패 시나리오마다 다음을 점검한다.

1. 이 시나리오가 실제로 발생할 가능성이 있는가?
2. 발생했을 때의 피해는 얼마나 큰가? (사용자가 포기할 정도인가, 살짝 불편한가?)
3. 프롬프트 안에 이걸 막을 장치가 있는가?
4. 없다면, 추가해야 하는가?

모든 시나리오를 방어할 수는 없다. 발생 확률과 피해의 곱이 큰 것부터 처리한다.

### 주의점

과도한 방어는 프롬프트를 경직시킨다. **모든 예외를 처리하려 하지 말고, 중요한 것만 처리**하라. 그리고 예외 처리는 "금지 조항 추가"보다 **"기본값 설정"** 쪽으로 하라. (원리 2 참조)

---

## 도구 4: 최소 호출 원칙

### 언제 쓰는가

프롬프트의 재사용성과 접근성을 높이고 싶을 때.

### 핵심 아이디어

사용자가 **가장 귀찮아서 할 법한 호출 방식**을 상정하고 프롬프트를 설계한다.

이상적인 사용자가 친절하게 모든 맥락을 제공하는 호출이 아니라, **"X 알려줘"** 한 줄만 던지는 경우를 기준점으로 삼는다. 이 호출에서 프롬프트가 여전히 작동해야 한다.

### 왜 이것이 중요한가

현실에서 사람은 귀찮음에 약하다. 프롬프트 호출이 복잡하면 잘 쓰지 않게 된다. 특히 반복 사용할 프롬프트일수록 **진입 비용**이 크다. 한 번 쓸 때마다 긴 맥락을 써야 한다면, 결국 쓰지 않게 된다.

반면 "X 알려줘"만으로도 쓸 만한 결과가 나오면, 사용자는 자주 쓰게 되고 점점 깊이 쓰게 된다. 맥락을 추가로 제공하면 결과가 더 좋아지는 **점진적 향상** 구조가 이상적이다.

### 실전 적용

프롬프트를 설계할 때 다음 세 가지 호출 케이스를 모두 시뮬레이션하라.

1. **최소 호출**: 필수 정보만 (예: 용어 하나만)
2. **중간 호출**: 약간의 맥락 추가 (예: 용어 + 관심 분야)
3. **최대 호출**: 모든 세부 맥락 (예: 용어 + 관심 분야 + 수준 + 배경 + 목적)

각 호출에서 나오는 결과의 차이가 **합리적으로 점진적**이어야 한다. 최소 호출에서 아예 못 쓸 결과가 나오거나, 최대 호출과 큰 차이가 없으면 설계에 문제가 있다.

### 주의점

최소 호출을 지원한다는 것은 AI가 많은 것을 **추론**해야 한다는 뜻이다. 추론이 틀릴 수도 있다. 이를 대비한 안전장치가 필요하다.

- 추론 결과를 내부적으로만 사용하고 응답에 명시하지 않기
- 추론이 갈릴 수 있는 경우(예: 다의어) 짧게 확인 질문
- 결과 말미에 "다른 맥락으로 원하면 알려달라" 류의 여지 남기기

## 도구 5: 레이어드 아웃풋

### 언제 쓰는가

응답이 길어질 때, 다양한 독자를 고려해야 할 때.

### 핵심 아이디어

결과물을 **모든 독자가 처음부터 끝까지 읽는다고 가정하지 마라**. 실제로는 이렇게 읽는다.

- 바쁜 사람: 요약만 읽고 닫는다
- 관심 있는 사람: 특정 섹션만 읽는다
- 진지한 사람: 전체를 읽는다

이 세 독자 모두에게 가치를 주는 출력은 **층위적으로 설계**된다.

### 층위적 출력의 구조

```
┌─────────────────────┐
│  초압축 요약                                                     │  ← 30초 독자용
│  (맨 위)                                                            │
├─────────────────────┤
│                                                                        │
│  본문                                                               │  ← 2~5분 독자용
│  (중간, 대부분)                                                │
│                                                                        │
├─────────────────────┤
│  확장 자료                                                        │  ← 10분+ 독자용
│  한 장 요약                                                       │
│  후속 제안                                                        │
│  (맨 아래)                                                        │
└─────────────────────┘
```

각 층에서 읽기를 멈춰도 가치가 있어야 한다.

### 실전 적용

출력 설계 시 다음 질문에 답하라.

1. **30초 독자가 얻어가야 할 핵심 메시지는 무엇인가?** → 이것이 상단 요약이 된다.
2. **2~5분 독자가 필요한 모든 정보는 무엇인가?** → 이것이 본문이 된다.
3. **진지한 독자가 더 깊이 파고들 수 있는 확장은 무엇인가?** → 이것이 하단이 된다.

각 층의 **분량 비율**도 중요하다. 경험적으로 5~15% (상단) / 70~85% (본문) / 10~20% (하단) 정도가 안정적이다. 상단이 너무 길면 요약의 기능을 잃고, 하단이 너무 길면 중심이 흐려진다.

### 주의점

레이어드 아웃풋은 내용이 **일정 길이 이상**일 때 의미가 있다. 짧은 응답에 억지로 상단 요약을 붙이면 부자연스럽다. 본문이 300자 미만이면 층위 구분 없이 통으로 쓰는 게 낫다.

---

## 도구 6: 제약을 자유로 뒤집기

### 언제 쓰는가

프롬프트가 경직되게 느껴질 때, "반드시/항상"이 많을 때.

### 핵심 아이디어

프롬프트 설계에서 자주 하는 실수가 **모든 경우에 대응하는 경직된 규칙**을 쓰는 것이다.

- "반드시 X해라"
- "항상 Y해라"
- "절대 Z하지 마라"

이런 규칙은 명확해 보이지만, 실전에서는 부작용을 낳는다. 규칙이 적용되면 안 되는 경우에도 적용되기 때문이다.

### 더 성숙한 접근

**"기본값 + 예외 규칙"** 구조를 쓴다.

- "원칙적으로는 이렇게 하되, 이런 경우엔 다르게 해라"
- "대부분의 경우 X하되, Y 조건에서는 Z한다"
- "기본값은 A이지만, 사용자가 명시하면 B로 전환한다"

### 실전 적용

경직된 규칙과 유연한 규칙의 차이를 예시로 보자.

**경직형:**

> "질문을 받으면 반드시 먼저 사용자의 배경을 물어봐라."

**유연형:**

> "원칙적으로 질문하지 말고 바로 답변해라. 단, 용어가 서로 다른 분야에서 같은 이름으로 쓰이는 다의어일 때만 맨 앞에서 한 줄로 확인해라."

유연형은 더 길지만 실전 품질이 훨씬 높다. 일반 사용자의 귀찮음을 줄이면서도 필요한 경우의 안전장치는 유지한다.

### 경직형을 유연형으로 전환하는 절차

1. 프롬프트에서 "반드시/항상/절대" 표현을 전부 찾는다.
2. 각 규칙에 대해 묻는다: "이 규칙이 적용되면 안 되는 경우가 있는가?"
3. 있다면 "기본값 + 예외" 형태로 재작성한다.
4. 없다면(정말로 예외 없이 적용되어야 한다면) 그대로 둔다.

대부분의 규칙은 생각보다 많은 예외를 가진다. 이 작업만으로도 프롬프트의 유연성이 크게 올라간다.

### 주의점

유연한 규칙은 AI에게 **판단의 여지**를 준다. 이 판단이 틀릴 수 있다. 그래서 다음을 병행하라.

- 예외 조건을 최대한 구체적으로 기술 ("다의어" 같은 모호한 단어보다 "물리학과 정보이론에서 같은 이름으로 쓰이는 경우" 같은 구체적 기술)
- 기본값이 "더 안전한 쪽"이 되도록 설계 (실패해도 큰 피해 없는 쪽)

# 3층: 실전 워크플로우

원리와 도구를 **실제 개발 과정**에 어떻게 배치할 것인가. 이 층은 "언제 뭘 할 것인가"를 다룬다.

워크플로우는 7개의 Phase로 나뉜다. Phase 0부터 6까지, 각 단계는 고유한 목적과 산출물을 가진다. 순서대로 진행하지만, Phase 4의 메타 점검 후에는 필요에 따라 이전 Phase로 되돌아갈 수 있다.

## Phase 0: 문제 설정

### 왜 이것이 가장 중요하고 가장 과소평가되는가

많은 사람이 Phase 0 없이 Phase 1로 바로 들어간다. "블록체인 알려주는 프롬프트를 만들자"는 생각이 들면 곧바로 프롬프트를 쓰기 시작한다.

이것의 문제는 **목표가 불명확하면 개선의 방향을 잡을 수 없다**는 것이다. 어떤 결과가 "좋은 것"인지 모르면, 두 개의 프로토타입 중 무엇이 더 나은지 판단할 수 없다. 그래서 개선이 표류한다.

Phase 0에서 시간을 써서 목표를 분명히 하는 것이 이후 모든 단계의 효율을 결정한다.

### Phase 0에서 답해야 할 질문들

**질문 1: 무엇을 위한 도구인가?**

- 일회성 작업인가 반복 작업인가?
- 나만 쓸 것인가 다른 사람도 쓸 것인가?
- 자주 쓸 것인가 가끔 쓸 것인가?

이 답에 따라 투자할 설계 에너지가 달라진다. 일회성 혼자용 프롬프트에 복잡한 구조는 과잉이다. 반복 사용될 공용 프롬프트에 허술한 구조는 부족이다.

**질문 2: 성공을 어떻게 판별할 것인가?**

이게 빠지면 개선의 방향 감각을 잃는다. "좋은 결과물"의 기준이 무엇인지, 최소 한두 개의 **검증 케이스**를 미리 생각해두라.

예를 들어 학습 프롬프트라면:

- "전혀 모르는 용어를 입력했을 때, 30분 내에 관련 대화를 따라갈 수 있는 이해가 생기는가?"
- "실무에서 쓰기 위한 용어를 입력했을 때, 추가 학습 방향이 구체적으로 도출되는가?"

이런 기준이 있어야 결과물을 평가할 수 있다.

**질문 3: 핵심 사용자는 누구인가?**

최소한의 페르소나 정의가 필요하다. (도구 1: 페르소나 다중화)

- 기본 수준(초보/중급/전문가)은 어느 쪽인가?
- 어떤 맥락에서 쓰게 될 것인가?
- 어떤 톤을 선호할 것인가?

완벽할 필요는 없다. 초기 가설로 시작하고, 이후 단계에서 조정한다.

### Phase 0 산출물

Phase 0을 마치면 다음 세 가지가 문서화되어 있어야 한다.

1. **목적 한 문장**: 이 프롬프트는 누가 언제 왜 쓰는가
2. **성공 기준 2~3개**: 어떤 결과가 나와야 성공인가
3. **핵심 사용자 페르소나**: 최소 한 명, 이상적으로는 세 명

이 문서는 짧아도 된다. 중요한 건 **있냐 없냐**다.

---

## Phase 1: 프로토타입

### 이 단계의 목적

**"돌아가는 것"을 만드는 것**이지, "좋은 것"을 만드는 것이 아니다.

많은 사람이 첫 버전에서 완벽을 추구하다가 진도가 안 나간다. Phase 1의 핵심 규율은 **완성도 40% 목표**다. 60%가 부족해도 좋다. 중요한 건 실행해볼 수 있는 형태가 되는 것이다.

### 왜 불완전한 프로토타입이어야 하는가

머릿속에서 프롬프트를 아무리 다듬어도, 실제 결과를 보기 전까지는 진짜 문제가 보이지 않는다. 실행해야만 드러나는 것들이 있다.

- AI가 예상과 다르게 해석하는 지점
- 사용자 입력과 프롬프트 구조 사이의 마찰
- 생각지 못한 빈 공간 (프롬프트가 다루지 않는 케이스)
- 잉여 지시 (필요 없어 보이는 부분)

이것들은 **실제 결과물 앞에서만 보인다**. 그래서 빨리 실행 가능한 형태로 만드는 것이 우선이다.

### Phase 1 가이드라인

1. **시간 제한을 둬라**: 30분~1시간 내로 끝낸다. 넘어가면 Phase 0으로 돌아가 목적이 불명확한 건 아닌지 점검한다.
2. **최소 구조만 갖춰라**: 역할, 핵심 지시, 출력 형식. 이 세 가지면 충분하다.
3. **예외 처리는 무시하라**: 이 단계에서는 정상 경로만 생각한다. 예외는 Phase 2에서 발견된다.
4. **참고 자료 보지 마라**: 다른 사람의 좋은 프롬프트를 보고 시작하면, 자기 문제에 맞는 설계가 아니라 남의 설계를 복제하게 된다.

### Phase 1 산출물

- 최소한 한 번 실행 가능한 프롬프트
- 그 프롬프트의 실행 결과 1~3개

---

## Phase 2: 진단

### 두 축으로 나눠 평가

프로토타입 결과를 두 축에서 독립적으로 평가한다. 이 구분이 핵심이다.

**내용 축 (What)**

- 빠진 것이 있는가?
- 얕은 것이 있는가?
- 틀린 것이 있는가?
- 깊이가 목적에 맞는가?

**행동 축 (How)**

- 불편한 점이 있는가?
- 예상과 다른 반응이 있는가?
- 사용에 마찰이 있는가?
- AI의 고정 반응이 부자연스러운가?

### 왜 둘을 분리해야 하는가

두 축은 **다른 수정 방향**을 요구한다.

- "깊이가 부족하다" → 프롬프트에 구조 추가, 섹션 확장
- "쓰기 불편하다" → 상호작용 방식 재설계, 입력 유연화

이걸 섞어서 한 번에 고치려 하면, 어떤 수정이 어떤 문제를 해결했는지 추적이 안 된다. 결과적으로 "이번 수정은 효과가 있었나 없었나"가 불분명해진다.

### 진단 결과의 번역

각 문제에 대해 **구체적인 개선 방향**을 한 문장으로 적어라.

**나쁜 번역:**

- "별로다"
- "뭔가 부족하다"
- "이해가 안 간다"

**좋은 번역:**

- "관계망 섹션에서 상위/하위 개념만 있고 형제 개념이 없다"
- "응답 시작 시 매번 질문하는 게 불필요하게 느껴진다"
- "실무 레벨 정보가 얕아서 표면적 이해에 그친다"

번역이 구체적일수록 다음 수정이 정확해진다. 번역이 모호하면 다음 버전도 방향 없이 표류한다.

### Phase 2 산출물

- 내용 축 문제 목록 (구체적 번역으로)
- 행동 축 문제 목록 (구체적 번역으로)
- 각 문제에 대한 우선순위 (심각도 × 빈도)

---

## Phase 3: 수정

### 한 번에 하나의 축, 한두 개 지점만

**원칙: 내용과 행동을 동시에 수정하지 않는다.** 무엇이 효과 있었는지 추적이 안 된다. 한 번에 한 축, 그것도 한두 개 지점만 바꾼다.

### 우선순위 판단

어느 축을 먼저 수정할지는 상황에 따라 다르다.

- **행동 축 문제가 심각하면 행동 먼저.** 쓰기 불편한 프롬프트는 아예 안 쓰게 된다. 내용이 아무리 좋아도 의미가 없다.
- **내용 축 결함이 치명적이면 내용 먼저.** 결과물이 쓸모없으면 쓸 이유가 없다.
- **보통은 행동 → 내용 순서가 수렴이 빠르다.** 행동이 안정되면 내용 평가가 정확해진다.

### 수정의 세 가지 레벨

문제의 성격에 따라 수정 레벨이 다르다.

**레벨 1: 표현 수정** (작은 수정)

- 문구를 더 명확하게 다듬기
- 예시 추가/제거
- 톤 조정

**레벨 2: 구조 수정** (중간 수정)

- 섹션 추가/제거
- 순서 변경
- 출력 형식 변경

**레벨 3: 성격 수정** (큰 수정)

- 상호작용 방식 전환 (대화형 ↔ 원샷형)
- 범용 ↔ 특화 전환
- 고정 구조 ↔ 동적 구조 전환

레벨 3은 Phase 4의 영역이다. Phase 3에서는 주로 레벨 1~2의 수정을 다룬다.

### Phase 3 산출물

- 수정된 프롬프트 (버전 번호 증가)
- 수정 로그 (무엇을 왜 바꿨는지)

버전 번호와 로그는 귀찮아 보이지만 중요하다. 3~4번 수정하고 나면 "내가 뭘 어떻게 바꿨지?"를 잊기 시작한다. 로그가 있어야 메타 점검이 가능하다.

---

## Phase 4: 메타 점검

### 왜 주기적 메타 점검이 필요한가

세부 조정(Phase 3의 레벨 1~2)만 반복하면 **국소 최적(local optimum)에 빠진다**. 한 방향에서 열심히 다듬고 있는데, 사실은 완전히 다른 방향이 더 나은 경우가 있다. 이걸 발견하려면 한 발 물러서야 한다.

### 메타 점검 타이밍

수정을 **2~3번 한 후**에 한 번씩 한다. 매번 하면 피로하고, 너무 늦게 하면 국소 최적에 깊이 빠진다. 2~3회 정도가 경험적으로 좋은 주기다.

### 메타 점검 질문

**핵심 질문:** "이 프롬프트의 근본 성격이 맞나?"

이 질문을 구체화하면:

- 이게 정말 대화형이어야 할까, 원샷형이 더 맞을까?
- 고정 구조여야 할까, 동적 구조가 더 맞을까?
- 상세해야 할까, 간결이 더 맞을까?
- 범용이어야 할까, 특화가 더 맞을까?
- 한 번의 호출로 완결이어야 할까, 여러 단계의 파이프라인이 더 맞을까?

이 질문들은 Phase 3의 세부 수정으로는 답이 안 나온다. **도구 2 (반전 테스트)**를 활용해 각 차원을 실제로 뒤집어보는 것이 가장 효과적이다.

### 메타 점검의 두 결과

**결과 A: "지금 성격이 맞다"** → 세부 수정을 계속한다. 단, 이제 그 선택이 **근거 있는 확신**이 되었다. 이전에는 "왠지 이게 맞아 보여서" 선택한 것이, 이제는 "반전 버전보다 이게 나은 이유가 X, Y, Z"로 정당화된다.

**결과 B: "바꿔야겠다"** → 큰 도약이 온다. 성격 자체를 바꾸는 수정을 한다. 이 경우 사실상 Phase 1로 돌아가 새로운 프로토타입을 만든다.

### Phase 4 산출물

- 현재 프롬프트 성격에 대한 명시적 판단
- (전환이 있다면) 새로운 방향성 기술

---

## Phase 5: 실전 테스트

### 사고실험의 한계

Phase 1~4는 본질적으로 **사고실험**이다. 머릿속 시뮬레이션과 몇 개의 테스트 케이스로 진행된다. 이 단계에서 프롬프트는 상당히 좋아져 있다. 하지만 아직 부족하다.

실제 사용에서만 드러나는 것들이 있다.

- 예상 못 한 입력 유형
- 예상 못 한 해석 편차
- 장기 사용 시의 피로감
- 다른 사람이 쓸 때의 오해

이것들은 사고실험으로 도달할 수 없다.

### Phase 5의 테스트 방법

**방법 1: 다양한 주제에 적용**

같은 프롬프트를 여러 주제에 써본다. 한 주제에서 잘 돌아가도 다른 주제에서는 약점이 드러날 수 있다. 학습 프롬프트라면 구체 기술, 추상 개념, 제도, 역사 등 다양한 범주를 시도한다.

**방법 2: 서로 다른 입력 스타일로 호출**

- 최소 호출 ("X 알려줘")
- 중간 호출 (약간의 맥락)
- 최대 호출 (모든 맥락)
- 비정상 호출 (모호함, 오타, 다른 언어 섞임)

(도구 4: 최소 호출 원칙 활용)

**방법 3: 반복 실행**

같은 입력으로 5~10번 실행한다. 결과가 얼마나 일관적인가? 편차가 허용 범위인가? 일관성은 프롬프트 성숙도의 중요한 지표다.

**방법 4: 다른 사람에게 써보게 하기**

이게 가장 강력한 테스트다. 만든 사람은 자기 의도를 알기 때문에 프롬프트의 빈 공간을 무의식적으로 채워 읽는다. 다른 사람은 빈 공간 앞에서 멈춘다.

### Phase 5에서 발견되는 것들

- **일관성 문제**: 같은 입력에 다른 결과
- **맥락 의존성**: 특정 주제에서만 잘 돌아감
- **입력 민감도**: 호출 방식이 조금만 달라도 품질이 급격히 변함
- **장기 사용 피로**: 한두 번은 괜찮은데 반복하면 싫증
- **이해 격차**: 만든 사람은 잘 쓰지만 다른 사람은 헤맴

이 중 하나라도 심각하게 발견되면 Phase 2로 돌아간다.

### Phase 5 산출물

- 실전 테스트 로그
- 발견된 이슈 목록
- 프롬프트의 **실제 역량 한계** 문서화

---

## Phase 6: 수렴 판단

### "언제 멈출 것인가"의 문제

"더 개선할 수 있나?"라는 질문에는 항상 **Yes**다. 무한히 개선할 수 있다. 하지만 언젠가는 멈춰야 한다. 계속 개선에만 매달리면 정작 **써먹지 못한다**.

중요한 건 질문을 바꾸는 것이다.

- **❌ 잘못된 질문**: "더 개선할 수 있나?"
- **✅ 올바른 질문**: "지금 상태로도 충분히 가치가 있는가?"

### 수렴 신호

다음 중 **두 개 이상**이 오면 릴리즈해도 된다.

1. **수정 효과 체감**이 미미하다. 바꿔도 결과 차이가 거의 없다.
2. **수정 방향에 대한 아이디어**가 고갈된다. 뭘 바꿔야 할지 잘 모르겠다.
3. **실전에서 큰 불편 없이 작동**한다. 대부분의 사용 시나리오를 커버한다.
4. **Phase 0의 성공 기준**을 만족한다. 초기에 설정한 검증 케이스를 통과한다.

### 릴리즈 이후의 개선

Phase 6의 "릴리즈"는 **개선의 종료**가 아니다. **사고실험의 종료, 실사용 데이터 수집의 시작**이다.

실제로 쓰면서 발견되는 개선점은 사고실험으로 찾은 것과 질이 다르다. 훨씬 구체적이고 현실적이다. 이 데이터를 모아 다음 버전(v2)으로 간다. 이 과정은 **장기 관점의 Phase 1~6 재시작**이다.

### 완벽주의의 함정

"충분히 좋음"이라는 개념에 저항감이 있는 사람이 많다. "더 좋게 할 수 있는데 왜 지금 멈추지?"라는 생각이 든다. 하지만 **완벽을 추구하면 영원히 릴리즈 못 한다**. 그리고 실사용 데이터 없이 하는 개선은 사고실험의 반복일 뿐이다.

**좋은 프롬프트는 완성되는 게 아니라 유지되는 것**이다. v1을 릴리즈하고, 쓰고, 배우고, v2를 만든다. 이 리듬이 완벽주의보다 훨씬 빠르게 좋은 프롬프트를 만든다.

### Phase 6 산출물

- 릴리즈 버전 프롬프트
- 현재 역량 및 한계 문서
- 다음 버전을 위한 개선 메모

## 함정 1: 설계에 무한 매몰

### 증상

- 프롬프트를 쓰다가 수정하고 다시 수정하기만 반복
- "아직 실행해볼 만큼 준비가 안 됐다"고 느낌
- 완벽한 사양을 먼저 만들고 구현으로 가려는 경향

### 원인

완벽주의, 또는 실패 두려움. 불완전한 프로토타입을 실행해서 "별로"라는 결과를 보고 싶지 않은 심리.

### 왜 이것이 함정인가

머릿속에서는 프롬프트가 얼마나 좋을지 알 수 없다. **실행해봐야만 진짜 문제가 보인다.** 설계에만 매달리면 진짜 문제를 끝까지 모른다.

### 대처

1. **시간 제한을 엄격히 지켜라.** Phase 1을 1시간으로 제한하고, 그 안에 반드시 실행 가능한 형태를 만든다.
2. **"불완전한 실행 > 완벽한 설계"를 명문화.** 이것을 개발 원칙으로 자신에게 선언한다.
3. **"첫 실행은 실패해도 된다"를 받아들여라.** Phase 1의 목적은 좋은 결과가 아니라 실행 가능한 형태다. 실패는 정상이다.

---

## 함정 2: 피드백 인플레이션

### 증상

- 매 버전마다 섹션이 추가됨
- 프롬프트가 점점 길어지고 복잡해짐
- "이것도 필요해, 저것도 필요해"라는 반응이 계속됨
- 어느 순간 프롬프트가 관리 불가능할 정도로 비대해짐

### 원인

모든 피드백을 반영하려는 성향. "빼는 것"보다 "더하는 것"이 심리적으로 쉽다.

### 왜 이것이 함정인가

프롬프트가 비대해지면:

- AI가 모든 지시를 일관되게 따르지 못함
- 사용자가 읽고 수정하기 힘듦
- 내부 모순이 생김 (섹션끼리 충돌)
- 본래 목적이 희석됨

### 대처

1. **버전당 핵심 개선 1~2개로 제한.** 모든 피드백을 한 번에 반영하지 말고, 중요도 상위 2개만 처리한다. 나머지는 다음 버전으로.
2. **"빼는 수정"을 의식적으로 도입.** 2~3번의 "더하는 수정" 후에는 반드시 "이 중 뺄 수 있는 건 없나?"를 점검한다.
3. **한 버전의 목표를 한 문장으로.** "v3의 목표는 X를 개선하는 것이다"라고 명확히 한다. 목표 외의 수정은 다음으로 미룬다.

---

## 함정 3: 자신의 선호를 보편 선호로 착각

### 증상

- "내가 불편하다"를 "모두가 불편하다"와 같이 여김
- 자신의 사용 스타일만 고려한 설계
- 다른 사람이 써봤을 때 예상외의 반응이 나옴

### 원인

인간의 자연스러운 인지 편향. 특히 혼자 개발하는 경우 이걸 자각하기 어렵다.

### 왜 이것이 함정인가

혼자 쓸 프롬프트는 이 편향이 문제없다. 오히려 자기 스타일에 최적화된 게 좋다. 하지만 다른 사람이 쓸 가능성이 있는 프롬프트에서는 심각한 문제가 된다. 만든 사람에게는 완벽한 프롬프트가 다른 사람에게는 쓸 수 없는 것이 될 수 있다.

### 대처

1. **도구 1 (페르소나 다중화) 의무화.** 혼자용이 아니라면 최소 세 개의 페르소나를 시뮬레이션한다.
2. **다른 사람에게 써보게 하기.** Phase 5에서 반드시 포함한다. 어렵다면 최소한 친구, 동료에게 부탁한다.
3. **"이 프롬프트를 처음 보는 사람"의 입장에서 재독.** 프롬프트를 쓴 직후가 아니라, 하루 지난 뒤에 다시 읽는다. 어느 정도 거리감이 생기면 자기 편향이 보인다.

---

## 함정 4: 메타 전환 회피

### 증상

- 비슷한 수정을 3번 이상 반복했는데도 만족스럽지 않음
- 세부 조정만 계속함
- "조금만 더 다듬으면 될 것 같은데"가 반복됨

### 원인

세부 조정이 근본 전환보다 **심리적으로 쉽다**. 구조를 뒤집는 것은 지금까지의 작업을 부정하는 느낌이 들어 회피하게 된다.

### 왜 이것이 함정인가

국소 최적(local optimum)에 갇힌다. 지금 방향에서는 최선이지만, 사실은 완전히 다른 방향이 더 좋은 경우가 있다. 이걸 발견 못 하면 영원히 중간 품질에 머문다.

### 대처

1. **신호 감지.** 같은 방향의 수정을 3번 이상 반복했는데도 만족스럽지 않으면, 반드시 Phase 4의 메타 점검으로 넘어간다. 이걸 자동 규칙으로 삼는다.
2. **도구 2 (반전 테스트) 의무 시행.** 메타 점검 시 반드시 적어도 한 차원을 뒤집어본다. 반전 버전을 실제로 작성하거나 시뮬레이션한다.
3. **"지금까지의 작업이 낭비가 아니다"를 받아들여라.** 반전 테스트에서 방향 전환이 결정되어도, 지금까지의 작업은 **이 방향의 한계를 이해하는 데 기여**했다. 낭비가 아니라 투자다.

---

## 함정 5: AI 출력 과신

### 증상

- AI가 내놓은 결과물을 그대로 받아들임
- 결과물의 그럴듯함과 정확성을 동일시함
- 특히 잘 모르는 분야의 결과를 무비판적으로 수용

### 원인

AI 출력이 자연스럽고 권위 있게 느껴짐. 실제로 많은 경우 정확하기도 함. 하지만 **그럴듯해 보이지만 틀린 경우**가 섞여 있다.

### 왜 이것이 함정인가

프롬프트가 잘 작동하는지 판단할 때 결과물의 정확성이 핵심인데, 이걸 검증 안 하면 **실제로는 부정확한 결과를 내는 프롬프트를 "좋다"고 평가**하게 된다. 특히 전문 분야에서 위험하다.

### 대처

1. **"아는 주제로 테스트" 의무화.** 자신이 이미 잘 아는 주제로 먼저 프롬프트를 테스트한다. AI가 얼마나 정확한지 판단할 수 있는 영역에서 먼저 검증해야, 모르는 영역에서의 출력을 신뢰할지 판단할 수 있다.
2. **불확실성 처리 장치 내장.** 프롬프트에 "확신이 낮거나 논쟁적인 내용은 표시해라", "시점이나 지역에 따라 다를 수 있는 정보는 명시해라" 같은 지시를 넣는다.
3. **여러 번 실행해서 편차 확인.** 같은 입력에 다른 결과가 나오면, 최소한 그 영역은 AI의 확신도가 낮다는 신호다.

---

## 함정 6: 재사용성 무시

### 증상

- 매번 조금씩 수정해서 쓰게 됨
- 같은 목적의 프롬프트를 여러 번 다시 씀
- 프롬프트가 특정 상황에만 맞게 경직되어 있음

### 원인

Phase 0에서 "반복 사용할 것인가"를 충분히 고려하지 않음. 또는 처음엔 일회성이라고 생각했는데 반복 쓰게 됨.

### 왜 이것이 함정인가

반복 사용할 프롬프트를 재사용성 없이 만들면:

- 매번 수정 비용 발생
- 버전 관리 혼란 (어느 게 최신인지 모름)
- 각 변형이 조금씩 다른 결함을 가짐
- 개선이 축적되지 않음

### 대처

1. **Phase 0에서 명시적으로 결정.** "이 프롬프트는 일회성인가 반복 사용인가?" 반복 사용이면 다음 단계에서 변수 분리를 고려한다.
2. **고정 부분과 가변 부분 분리.**

```
## 고정 부분 (변경 불필요)
[역할, 구조, 출력 형식 등]

## 가변 부분 (호출마다 입력)
[학습 대상, 맥락, 수준 등]
```

3. **최소 호출 원칙 적용.** 가변 부분을 최소화한다. 호출 시 입력해야 할 것이 적을수록 재사용 빈도가 높아진다.

---

## 함정 7: 방법론 그 자체에 중독

### 증상

- 모든 프롬프트에 풀스펙 방법론 적용
- 간단한 일회성 작업에도 Phase 0부터 시작
- 방법론을 따르느라 본래 하려던 작업이 지연

### 원인

방법론의 가치를 과잉 일반화. "이 방법론이 좋다"가 "항상 이 방법론을 써야 한다"로 왜곡됨.

### 왜 이것이 함정인가

방법론은 **복잡한 작업의 혼란을 줄이기 위한 것**이다. 단순한 작업에 적용하면 오히려 오버헤드만 생긴다. 30초면 끝날 일을 30분 걸리게 만든다.

### 대처

1. **시작 시 규모 판단.** "이게 정말 풀스펙 방법론이 필요한 프롬프트인가?"를 먼저 묻는다.
2. **프롬프트의 티어 분류.**

| 티어 | 용도             | 방법론 적용        |
| ---- | ---------------- | ------------------ |
| T1   | 일회성, 간단     | 없음, 그냥 쓴다    |
| T2   | 반복, 개인용     | Phase 0~3만        |
| T3   | 반복, 공용       | 풀스펙 적용        |
| T4   | 제품/서비스 내장 | 풀스펙 + 추가 검증 |

3. **방법론 자체도 도구일 뿐.** 방법론을 따르는 것이 목적이 아니라, 좋은 프롬프트를 만드는 것이 목적이다. 목적에 봉사하지 않는 단계는 건너뛴다.

## 실전에서의 리듬 — 요약

프롬프트 개발을 할 때 몸에 배야 할 리듬이 있다.

### 거시 리듬: 수렴 사이클

```
문제 정의 → 빨리 만들기 → 실행 → 진단 → 수정
        ↓
    2~3회마다 메타 점검
        ↓
    실전 테스트 → 수렴 판단
```

이 사이클을 한 번 돌면 프롬프트가 한 단계 성숙한다. 여러 번 돌면 실용 가능한 수준에 도달한다.

### 미시 리듬: 한 번의 수정

```
1. 진단: 이번에 고칠 건 무엇인가? (한 축, 한두 지점)
2. 수정: 어떻게 고칠 것인가? (최소 변경)
3. 기록: 무엇을 왜 바꿨는가? (로그)
4. 실행: 바꾼 게 효과가 있는가? (검증)
```

이 리듬이 있으면 수정이 체계적이고, 효과 추적이 가능하고, 롤백도 쉽다.

## 마지막 원칙: "충분함"의 감각

방법론은 결국 **충분함의 감각**을 기르는 것이다.

- 설계가 **충분히** 되었는가? (더 해도 소용없을 정도)
- 수정이 **충분히** 되었는가? (효과가 미미해질 정도)
- 테스트가 **충분히** 되었는가? (대부분 케이스를 커버)

"충분함"은 "완벽함"이 아니다. 완벽을 추구하면 영원히 도달 못 하고, 충분함은 실제 도달 가능한 지점이다.

좋은 프롬프트 개발자는 완벽주의자가 아니라 **"충분히 좋음"을 판별할 줄 아는 사람**이다. 그리고 그 판별 능력은 경험과 성찰로 기른다. 이 방법론은 그 경험과 성찰의 틀을 제공할 뿐이다.

## 부록: 빠른 참조 카드

### 프롬프트 개발을 시작할 때

- [ ] Phase 0: 목적·성공기준·페르소나 정의했는가?
- [ ] 프롬프트의 티어는? (T1 일회성 / T2 개인 반복 / T3 공용 / T4 제품)
- [ ] 티어에 맞는 방법론 깊이를 정했는가?

### 프로토타입을 만들 때

- [ ] 30분~1시간 제한을 둘 수 있는가?
- [ ] 최소 구조(역할, 핵심 지시, 출력 형식)만 갖췄는가?
- [ ] 완성도 40%를 목표로 하고 있는가?

### 진단할 때

- [ ] 내용 축과 행동 축을 분리했는가?
- [ ] 문제를 구체적 문장으로 번역했는가?
- [ ] 우선순위를 매겼는가?

### 수정할 때

- [ ] 한 번에 한 축, 한두 지점만 수정하는가?
- [ ] 수정 로그를 남기는가?
- [ ] 버전 번호를 올리는가?

### 메타 점검할 때 (2~3회 수정 후)

- [ ] "근본 성격이 맞나?"를 물었는가?
- [ ] 반전 테스트를 해봤는가?
- [ ] 판단 근거가 명확한가?

### 실전 테스트할 때

- [ ] 다양한 주제에 적용했는가?
- [ ] 여러 입력 스타일로 호출했는가?
- [ ] 반복 실행으로 편차를 봤는가?
- [ ] 다른 사람에게 써보게 했는가?

### 수렴 판단할 때

- [ ] Phase 0의 성공 기준을 만족하는가?
- [ ] 수정 효과가 미미해졌는가?
- [ ] 실전에서 큰 불편 없이 작동하는가?
- [ ] "충분히 좋다"라고 말할 수 있는가?

## 맺음말

프롬프트 개발은 글쓰기가 아니라 **설계**다. 설계는 반복과 수렴의 과정이다.

이 문서에서 제시한 방법론은 그 반복과 수렴을 체계화하려는 시도다. 하지만 방법론은 도구일 뿐, 목적이 아니다. 결국 중요한 것은:

**"충분히 좋은 프로토타입을 빨리 만들고, 실행하여 문제를 드러내고, 체계적으로 수정하고, 주기적으로 큰 그림을 다시 보고, 실전으로 검증하여, 적절한 시점에 수렴한다."**

이 리듬을 몸에 익히면, 어떤 새로운 프롬프트 개발 상황에서도 길을 잃지 않는다. 구체적인 기법들은 경험과 함께 축적되고, 때로는 이 방법론을 넘어서는 자기만의 변형도 만들게 된다. 그것이 이 방법론의 궁극적 목적이다.

방법론은 **디딤돌**이다. 언젠가는 딛고 올라서서 넘어가는 것이다.
