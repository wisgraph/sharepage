> **동그라미재단 AI Academy 3기 — 바이브코딩**

## 이 교재의 포지션

> **여러분은 개발자가 되는 게 아닙니다.** > **AI와 함께 서비스를 만드는 사람**이 되는 거예요.

AI는 코드를 쓸 수 있어요. 하지만 **어디에 뭘 만들지, 왜 그렇게 만들지, 지금 잘 만들어지고 있는지**를 판단하는 건 사람의 몫이에요. 그래서 필요한 건 **코딩 실력**이 아니라 **어휘력**이에요.

# Part 1. 왜 AI 시대에 아키텍처 어휘가 더 중요한가

## 비트겐슈타인의 명제

> **"내 언어의 한계는 내 세계의 한계다."**
> — 비트겐슈타인, 『논리철학논고』 5.6

이 명제를 AI 시대로 옮기면:

> **내가 쓸 수 있는 어휘의 한계 = 내가 AI와 소통할 수 있는 한계**

어휘 바깥의 모든 결정은 **나도 모르게 AI에게 통째로 맡기게 됩니다.**

## 실무 출신 임원 vs 낙하산 임원

|                     | 실무 임원              | 낙하산 임원      |
| ------------------- | ---------------------- | ---------------- |
| 회사가 잘 돌아갈 때 | 어디가 효율적인지 본다 | "잘되고 있군"    |
| 회사가 망가질 때    | 어디가 문제인지 짚는다 | "능력 바깥의 일" |
| 결정 근거           | 언어로 설명 가능       | 감에 의존        |

**AI와 일할 때도 똑같아요.** 어휘가 없으면 AI가 무엇을 하고 있는지 평가할 방법 자체가 없어져요.

## AI 구조 빈곤 — 왜 거의 필연인가

AI는 **공개된 방대한 코드**를 학습했어요. 그 중 어떤 코드는 아키텍처가 잘 짜여 있고, 어떤 코드는 한 파일에 다 몰아넣은 방식이에요. **AI는 양쪽 패턴 모두 능숙하게 재현**할 수 있어요.

문제는 프로젝트가 커질 때예요.

```merlight
flowchart TD
    Learned["📚 AI가 학습한 것<br>= 다양한 코드 전부"]
    Context["👁️ AI가 볼 수 있는 것<br>= 내 컨텍스트만<br><i>← 여기가 한계</i>"]
    Do["🔧 AI가 하는 일<br>= 기존 구조에 새 의도를 구겨넣기"]
    Result["💥 결과<br>중복 의도 / 중복 코드 / 빈약한 아키텍처"]

    Learned --> Context
    Context --> Do
    Do --> Result

    style Learned fill:#e8f5e9,stroke:#4CAF50
    style Context fill:#fff3e0,stroke:#FF9800
    style Do fill:#fff3e0,stroke:#FF9800
    style Result fill:#ffcdd2,stroke:#f44336
```

이 과정이 **관리 없이 방치하면 거의 필연적인 귀결**이에요.

## 오늘의 앵커 한 줄

> **아키텍처는 사람이 미리 잡고, 드리프트는 도구로 탐지한다.**

- **"미리 잡기"** = 여러분이 AI에게 방향을 지시할 때 쓸 어휘 (Part 2, 3)
- **"도구 탐지"** = intent-architect 같은 진단 도구 (Part 4)

# Part 2. 구조 없는 코드가 어떻게 망가지는가

게임을 만든다고 상상해보세요. **기사(Knight)**와 **마법사(Mage)**가 있어요.

## Step 1. 깨끗한 시작

```merlight
flowchart TD
    Knight["⚔️ <b>Knight</b> (기사)<br>━━━━━━━━━━━<br>• 체력 hp<br>• 마나 mp<br>• slash() — 베기<br>• stab() — 찌르기"]

    style Knight fill:#e8f5e9,stroke:#4CAF50,stroke-width:2px
```

**이름만 봐도 기사**예요. 근접 전투 전문가. 깔끔한 상태.

## Step 2. 오염의 시작

요구사항: "파이어볼 기능 넣어주세요."

AI에게 "Knight에 파이어볼 추가해줘"라고 말하면 AI는 시키는 대로 해요.

```merlight
flowchart TD
    Knight["⚔️ <b>Knight</b> (기사)<br>━━━━━━━━━━━<br>• 체력 hp<br>• 마나 mp<br>• slash() — 베기<br>• stab() — 찌르기<br>• 🔥 fireball() ← <i>이상한 일이 일어났다</i>"]

    style Knight fill:#fff3e0,stroke:#FF9800,stroke-width:2px
```

**기사가 마법을 씁니다.** 이 순간부터 오염이 시작돼요.

## Step 3. 버그 확산

나중에 **마법사(Mage)** 캐릭터를 새로 만들었어요.

```merlight
flowchart LR
    Knight["⚔️ <b>Knight</b><br>━━━━━━━<br>• 🔥 fireball()<br><i>(여전히 여기)</i>"]
    Mage["🧙 <b>Mage</b> (새로 만듦)<br>━━━━━━━<br>• heal() — 치유<br>• ❌ fireball() 없음"]

    style Knight fill:#ffcdd2,stroke:#f44336,stroke-width:2px
    style Mage fill:#e8f4f8,stroke:#2196F3,stroke-width:2px
```

**마법사가 마법을 못 쏩니다. 기사가 쏴요.** 버그 발생.

## Step 4. 스파게티 탄생

급한 마음에 if문으로 땜질:

```
만약 사용자가 마법사다 → mage.fireball() 부르자
아니면 → knight.fireball() 부르자 (아직 여기 있음)
```

Mage에도 fireball()을 새로 만들었고, Knight에서 지우는 건 무서워서 두고...

**같은 기능이 두 곳에 존재**. 이걸 **스파게티 코드**라고 불러요.

## AI가 스파게티를 재생산한다

이게 진짜 핵심이에요.

> **AI는 여러분의 코드를 읽고 같은 수준의 코드를 만듭니다.**

Knight에 fireball이 있는 걸 본 AI는 "아, 이 프로젝트에선 캐릭터에 직접 스킬을 넣는구나" 하고 판단해요. 다음에 "아이스볼 추가해줘"라고 하면? Knight에도 Mage에도 iceball을 넣어버립니다. 스파게티가 두 배.

**처음에 구조를 잘 잡는 게 바이브코딩에서 특히 중요해요.**

## 이 문제의 이름 — 단일 책임 원칙 (SRP)

이 상황에 이름을 붙여봅시다.

> **단일 책임 원칙 (Single Responsibility Principle, SRP)**:
> 하나의 클래스는 하나의 책임만 져야 한다.

- Knight는 근접 전투만
- Mage는 마법만
- 파이어볼은 Mage에만

**AI에게 할 수 있는 말**:

```
"이 클래스가 너무 많은 일을 해. 단일 책임 원칙에 맞게 분리해줘."

"Knight에 마법 관련 메서드가 섞여 있어.
마법은 Mage 클래스로 옮기고, Knight는 근접 전투만 담당하게 해줘."
```

원칙의 이름을 알면 AI에게 정확하게 지시할 수 있어요.

# Part 3. 어휘 사전 — AI에게 지시할 때 쓰는 이름들

이 파트에 나오는 어휘 **33개**를 알면, AI와의 소통이 **낙하산에서 실무파로** 바뀌어요.

---

## 3-1. 프로젝트 규모별 아키텍처 템플릿

**"방 배치"를 어떻게 잡을 것인가**의 3가지 표준입니다.

| 규모                         | 프로젝트 예시                          | 폴더 구성                                                 |
| ---------------------------- | -------------------------------------- | --------------------------------------------------------- |
| **Small** (1~2 기능)         | 랜딩페이지, 계산기, 퀴즈, 포트폴리오   | `views/` + `utils/`                                       |
| **Medium / MVC** (3~5 기능)  | 투두 앱, 간단 쇼핑몰, 게시판           | Small + `models/` + `controllers/` + `services/`          |
| **Large / 계층형** (6+ 기능) | 에디터, 협업 툴, 복잡한 대시보드, SaaS | Medium + `stores/` + `repositories/` + `api/` + `events/` |

**핵심 규칙**: 프로젝트가 복잡해질수록 폴더가 **더해지는** 구조예요. Small에서 시작해서 커지면 Medium, 더 커지면 Large.

**중요한 오해 방지**: Large는 백엔드 전용이 아니에요. 복잡한 프론트엔드(에디터, 협업 툴, 대시보드)도 **Large 구조**가 필요해져요. 백엔드냐 프론트엔드냐가 아니라 **복잡도**가 tier를 결정합니다.

> **AI에게 할 말** (기획서 첨부하고):
> "이 기획서 기반으로 프로젝트 구조를 잡아줘. 최종 기능 목록 전체를 고려해서 확장 가능한 구조로. 아직 안 만들 기능도 폴더는 미리 만들어두고, 각 폴더에 `README.md`로 '이 폴더의 역할'을 적어줘. '오버엔지니어링'이라고 줄이지 마."

## 3-2. 폴더 어휘 사전 — 10개 이름

각 폴더가 **어떤 역할**을 담는지 + **Knight/Mage 게임**으로 구체 예시 + **언제 필요**한가.

| 폴더            | 하는 일          | 🎮 Knight/Mage 예시                                    | 언제 필요한가              |
| --------------- | ---------------- | ------------------------------------------------------ | -------------------------- |
| `models/`       | 데이터 구조 정의 | Knight, Mage 클래스 자체                               | 데이터 모양이 정해질 때    |
| `views/`        | 화면에 보여줄 것 | 전투 화면, HP 바, 캐릭터 카드                          | 거의 모든 프로젝트         |
| `controllers/`  | 흐름 제어        | "공격 버튼 눌림 → 어떤 함수 실행할지" 판단             | 상호작용 있는 순간부터     |
| `services/`     | 비즈니스 로직    | 데미지 공식, 크리티컬 확률, 레벨업 조건                | 규칙·계산이 복잡해질 때    |
| `stores/`       | 전역 상태        | 현재 파티, 파티 HP 합계, 인벤토리                      | 여러 화면이 같은 상태 공유 |
| `repositories/` | 저장/조회 창구   | 세이브 파일에서 캐릭터 불러오기·저장                   | DB·외부 저장소 쓸 때       |
| `api/`          | 외부 통신        | 온라인 랭킹 서버, 업데이트 체크                        | 외부 API 연동              |
| `events/`       | 이벤트 버스      | "공격 발생!" → 애니메이션+사운드+로그+퀘스트 동시 반응 | 한 사건에 여러 반응        |
| `middleware/`   | 요청 가로채기    | 공격 전 "MP 충분한가?" 체크, 동작 로그                 | 공통 전처리 필요           |
| `utils/`        | 공통 도구 함수   | 데미지 랜덤 계산, 주사위, 시간 포맷                    | 거의 모든 프로젝트         |

**웹 서비스로 바꿔도 같은 구조**예요:

- `models/` → 사용자·상품·주문 정의
- `stores/` → 로그인 상태, 장바구니
- `services/` → 할인율 계산, 배송비 규칙
- `repositories/` → Supabase 호출 모음
- `middleware/` → 로그인 체크, 요청 로깅

> **AI에게 할 말**:
> "앞으로 이 프로젝트에서 기능추가 및 확장성을 고려할때,
> models, views, controllers, services, stores, repositories, api, events, middleware, utils의 계층을 사용해야할지 검토해주고 조언해줘."

## 3-3. 데이터 흐름 — 방들 사이의 배관

폴더를 나누는 것만으로는 부족해요. **방들 사이의 연결 방향**도 규칙이 있어야 해요.

### 정상 흐름

```merlight
flowchart TD
    User["👤 사용자"]
    V["views/<br>화면"]
    C["controllers/<br>흐름 제어"]
    S["services/<br>계산·규칙"]
    R["repositories/<br>DB 창구"]
    ST["stores/<br>상태"]

    User -->|"클릭"| V
    V -->|"이벤트"| C
    C -->|"요청"| S
    S -->|"조회"| R
    S -->|"결과"| ST
    ST -->|"화면 갱신"| V
    V -->|"표시"| User

    style V fill:#e8f4f8,stroke:#2196F3
    style ST fill:#e8f4f8,stroke:#2196F3
    style C fill:#fff3e0,stroke:#FF9800
    style S fill:#fff3e0,stroke:#FF9800
    style R fill:#e8f5e9,stroke:#4CAF50
```

**한 방향으로 흘러요.** 사용자 → View → Controller → Service → Repository → 결과가 다시 Store → View로 돌아오는 순환. 화살표가 **규칙적**이라는 점이 핵심이에요.

### 스파게티 흐름 — 두 가지 얼굴

구조가 없을 때 나타나는 **두 가지 극단**이 있어요. 둘 다 나쁘지만 모양이 달라요.

#### ① 몰아넣기 — 파일 하나에 다 넣은 경우

```merlight
flowchart TD
    User["👤 사용자"]
    App["app.js<br>(모든 것이 여기에)"]

    User -->|"클릭"| App
    App -->|"직접 DB 접근"| App
    App -->|"직접 화면 그림"| App
    App -->|"직접 계산"| App
    App -->|"직접 API 호출"| App
    App -->|"결과 표시"| User

    style App fill:#ffcdd2,stroke:#f44336
```

**모든 화살표가 한 파일을 가리켜요.** 어디를 고치면 어디가 터질지 예측 불가. 이게 **주방과 화장실이 결합된 집** 비유예요. 초기 프로토타입에서 흔해요.

#### ② 진짜 스파게티 — 파일은 여럿인데 얽힌 경우

파일은 나눴는데 **누가 누구를 부르는지 규칙이 없을 때**가 더 무서워요.

```merlight
flowchart TD
    User["👤 사용자"]
    UI["ui.js<br>화면"]
    Knight["knight.js<br>기사"]
    Mage["mage.js<br>마법사"]
    Battle["battle.js<br>전투"]
    Save["save.js<br>저장"]
    API["api.js<br>서버"]

    User --> UI
    UI --> Knight
    UI --> API
    Knight --> UI
    Knight --> Mage
    Knight --> Save
    Mage --> Knight
    Mage --> Battle
    Battle --> Knight
    Battle --> Save
    Battle --> UI
    Save --> Battle
    Save --> API
    API --> UI
    API --> Save

    style Knight fill:#ffe0b2,stroke:#f57c00
    style Mage fill:#ffe0b2,stroke:#f57c00
    style Battle fill:#ffcdd2,stroke:#d32f2f
    style Save fill:#ffcdd2,stroke:#d32f2f
    style UI fill:#ffcdd2,stroke:#d32f2f
    style API fill:#ffcdd2,stroke:#d32f2f
```

화살표가 사방팔방으로 뻗어 있어요. `knight.js`가 `mage.js`를 부르고, `mage.js`가 다시 `battle.js`를 부르고, `battle.js`가 또 `knight.js`로 돌아와요. **순환 고리**가 여러 개 생겼어요. `ui.js`가 서버(`api.js`)를 직접 부르기도 하고요.

증상은 이렇게 나와요:

- 🔴 **수정 공포증** — `knight.js` 하나 고쳤는데 저장이 깨짐. 어디서 연결됐는지 모름
- 🔴 **AI가 같은 기능을 세 번 만듦** — "힐 추가해줘" 했더니 AI가 knight, battle, save 세 곳에 비슷한 로직을 각각 심어버림
- 🔴 **디버깅 불가** — "HP가 왜 안 줄지?" 추적하려는데 Battle → Save → Knight → Battle 도돌이표

> 💡 **핵심 차이:**
>
> - **몰아넣기**는 "파일을 아예 안 나눈 상태" — 초보 프로토타입
> - **스파게티**는 "파일은 나눴는데 방향이 없는 상태" — AI 바이브코딩 6개월차에 가장 흔한 모양
>
> 폴더 만든다고 해결되는 게 아니에요. **누가 누구를 부르는지**(방향 규칙)가 잡혀야 진짜 정리된 거예요. 이건 수강생이 직접 외울 영역이 아니라, **intent-architect 같은 도구가 자동으로 점검**해줘요 (Part 4-3 참고).

### 흐름 규칙의 의미

> **흐름의 방향을 설계하는 것이 아키텍트의 영역입니다.**

- `views/`가 직접 `repositories/`를 부르면 안 돼요 — `controllers/`를 거쳐야 해요
- `models/`는 다른 폴더를 부르지 않아요 — 데이터 정의만 하는 방
- `utils/`는 어디서든 쓸 수 있어요 — 공용 도구

이 규칙은 수강생이 직접 지키는 게 아니에요. **intent-architect 같은 도구가 자동으로 점검**해요 (Part 4-3 참고).

## 3-4. 패턴·기법 어휘 사전 — 8개 상황 매핑

자주 마주치는 **상황 → 이름 → AI 프롬프트**를 매트릭스로 정리:

| 패턴·기법                 | 어떤 상황                                                           | AI에게 할 말                         |
| ------------------------- | ------------------------------------------------------------------- | ------------------------------------ |
| **SSOT** (진실은 한 곳에) | 같은 데이터가 여러 곳에 흩어져 동기화 사고 (로그인 상태가 여기저기) | "SSOT로 ○○ 상태를 한 곳에 모아줘"    |
| **FSM** (유한 상태 기계)  | 모드가 여럿, 동시에 켜지면 안 됨 (연필/지우개, idle/편집)           | "FSM으로 모드 관리해줘"              |
| **Command**               | Undo/Redo 필요, 키보드 단축키, 동작 히스토리                        | "Command 패턴으로 undo/redo 넣어줘"  |
| **Event Bus**             | 한 사건에 여러 곳 동시 반응 (저장 시 알림+로그+싱크)                | "Event Bus로 느슨하게 연결해줘"      |
| **Repository**            | DB를 나중에 바꿀 수 있게 분리 (Supabase → Firebase)                 | "Repository 패턴으로 DB 접근 감싸줘" |
| **Middleware**            | 공통 처리 (인증·로깅)를 요청마다                                    | "인증 미들웨어로 분리해줘"           |
| **Debounce**              | 이벤트 폭발 (검색창 자동완성)                                       | "검색창에 debounce 300ms 걸어줘"     |
| **Dispatch Table**        | `if/else` 체인 반복 (타입별 분기)                                   | "Dispatch Table로 매핑해줘"          |

> **용어 주의**: 위 8개는 엄밀히 따지면 모두 "디자인 패턴"은 아니에요. Middleware는 아키텍처 메커니즘, Debounce는 성능 기법, Dispatch Table은 프로그래밍 기법. **실용적으로 AI 지시 어휘로 함께 묶었을 뿐**이에요. SW 엔지니어 만날 때는 "패턴과 기법"을 구분하면 정확.

### 더 있는 어휘들 (Tier 2)

이름만 알면 충분, AI에게 이 프로젝트의 기능들을 토대로 어떤 디자인 패턴이 필요한지 물어보기만 해도 좋은 조언을 얻을 수 있습니다.

| 이름              | 핵심 아이디어                                                                             |
| ----------------- | ----------------------------------------------------------------------------------------- |
| **Observer**      | 데이터 변경 시 구독자(신호받아서 처리해야하는 코드들)에게 자동 알림 (stores/의 작동 원리) |
| **Singleton**     | 앱 전체에 하나만 존재 (현재 로그인 유저, 앱 설정)                                         |
| **Strategy**      | 알고리즘 교체 가능 (결제 수단: 카드/카카오페이/토스)                                      |
| **Adapter**       | 외부 API를 내 스타일로 감싸기                                                             |
| **Optimistic UI** | 서버 응답 기다리지 않고 UI 먼저 반응                                                      |
| **Orchestrator**  | 긴 함수를 지휘자 + 연주자로 분해                                                          |
| **Lazy Loading**  | 필요할 때만 로드 (이미지 많은 페이지)                                                     |

## 4-1. AI에게 구조적으로 지시하기

### 나쁜 지시 vs 좋은 지시

| ❌ 나쁜 지시             | ✅ 좋은 지시                                                      | 왜?           |
| ------------------------ | ----------------------------------------------------------------- | ------------- |
| "파이어볼 기능 추가해줘" | "Mage 클래스에 fireball 메서드를 추가해줘. Knight는 건드리지 마." | **위치** 지정 |
| "코드 정리해줘"          | "이 함수가 너무 많은 일을 해. 단일 책임 원칙에 맞게 분리해줘."    | **원칙** 지정 |
| "에러 고쳐줘"            | "이 에러 메시지 보여줄게: [붙여넣기]. 원인 분석해줘."             | **근거** 제공 |

### 프롬프트 템플릿 5종

**① 구조 잡기 (프로젝트 초반)**

```
"이 기획서 기반으로 프로젝트 구조를 잡아줘.
Medium/MVC 규모로 가고, 아직 안 만들 기능도 폴더는 미리 만들어둬.
각 폴더에 README.md로 '이 폴더의 역할'을 적어줘."
```

**② 기능 추가 (위치 명시)**

```
"○○ 기능을 추가해줘.
- models/에 데이터 구조
- services/에 로직
- views/에 화면
이렇게 분리해서 넣어줘."
```

**③ 리팩토링 (원칙 기반)**

```
"이 파일에 [A 기능]과 [B 기능]이 섞여 있어.
단일 책임 원칙에 맞게 분리해줘.
[A]는 [파일명A]로, [B]는 [파일명B]로."
```

**④ 패턴 적용 (이름 기반)**

```
"○○ 기능에 undo/redo를 넣고 싶어.
Command 패턴으로 구현해줘."
```

**⑤ 디버깅 (증상 기반)**

```
"이 에러가 나요:
[에러 메시지 그대로 붙여넣기]
원인을 분석해줘. 추측 말고 로그를 찍어서 확인해줘."
```

### 지시 원칙 3가지

1. **위치 명시** — "어느 폴더, 어느 클래스"를 지정
2. **원칙 명시** — SRP·패턴 이름 등 어휘로 방향 제시
3. **증거 제공** — 에러 메시지·상황 설명을 해석하지 말고 원본 그대로

## 4-2. Git — 코드의 세이브 포인트

수업에서 Knight가 오염되는 걸 봤죠. 그 전에 세이브했으면 돌아갈 수 있었어요.

### 필요한 5가지 명령

| 명령                     | 뜻                                   | 언제                    |
| ------------------------ | ------------------------------------ | ----------------------- |
| `git init`               | Git 시작 (처음 한 번)                | 프로젝트 시작 시        |
| `git add .`              | 변경사항 담기                        | 저장하기 전             |
| `git commit -m "메시지"` | 세이브 포인트 생성                   | 잘 되는 시점마다        |
| `git log`                | 기록 보기                            | 언제 뭘 저장했는지 확인 |
| `git checkout .`         | 마지막 세이브로 복귀 (미저장 변경만) | 망가뜨렸을 때           |

### 최초 세팅의 함정 — `.gitignore`

`git init` 바로 다음에 해야 할 일: **`.gitignore` 파일 만들기**.

이 파일은 "Git이 무시할 파일 목록"이에요. 안 만들면:

- **`.env`** (API 키·비밀번호)가 GitHub에 공개되어 털림
- **`node_modules/`**(의존성 수만 개 파일) 때문에 저장소 폭발
- **`.DS_Store`** 같은 잡파일까지 추적

> **AI에게 할 말**:
> "내 프로젝트에 `.gitignore` 파일을 만들어줘. Node.js / React 기준 기본 무시 목록 넣고, `.env`와 `.env.local`도 포함해줘."

# Part 5. 전체 지도 — 오늘 배운 게 아키텍처의 어디인가

지금까지 배운 **폴더 구조 + 패턴 어휘**로 아키텍처를 "다 배운 건가?" 하는 질문에 정직하게 답할게요.

> **아니지만, AI와 소통할 때 실제로 쓰는 어휘의 80%는 맞습니다.**

## 아키텍처의 5개 층위

|   #   | 층위                          | 질문                     |    이 수업     |
| :---: | ----------------------------- | ------------------------ | :------------: |
| **1** | **폴더 구조 / 모듈 경계**     | 어디에 둘 것인가?        |  ✅ Part 3-2   |
| **2** | **패턴 / 기법**               | 어떻게 묶여 움직이나?    |  ✅ Part 3-4   |
| **3** | **데이터 흐름 / 의존성 방향** | 누가 누구를 불러도 되나? | ⚠️ 도구가 탐지 |
| **4** | **경계·계약 (Interface)**     | 뭘 공개하고 뭘 숨기나?   |    ❌ 심화     |
| **5** | **비기능적 요구 (성능·보안)** | 얼마나 빠르고 안전해야?  |   ❌ 5회차+    |

## 각 층위가 뭘 의미하나

**1. 폴더 구조** — "방 배치." 오늘 다뤘음.

**2. 패턴** — "가구 배치." 오늘 다뤘음.

**3. 데이터 흐름** — "배관·전기 배선." 폴더가 있어도 이 방향 규칙이 깨지면 여전히 스파게티. intent-architect가 대신 봐줘요.

**4. 경계·계약** — "방 출입 규칙." 각 클래스가 외부에 뭘 공개할지. 프로젝트 커지면 배워요.

**5. 비기능적 요구** — "난방·보안." 성능, 인증·권한, 보안 등. 사용자 트래픽 생기면 중요해져요.

## 왜 3회차는 1~2번만 다루는가

**(1) AI 소통 어휘의 80%** — 일상 지시의 대부분이 이 두 층위에 해당.

**(2) 3번은 도구가 커버** — intent-architect 보고서가 "이 방향 위반했어요"를 알려줘서, 직접 설계 안 해도 됨.

**(3) 4~5번은 실제 사용자 있어야 의미** — MVP 단계에서는 시기상조. 5회차 이후.

## 1~2번만 잘 잡아도 스파게티 확률 급감

5개 층위가 동시에 무너지면 진짜 스파게티:

| 안 지켜지면         | 증상                                      |
| ------------------- | ----------------------------------------- |
| 1. 폴더 없음        | 모든 코드가 한 파일에                     |
| 2. 패턴 없음        | 복붙 코드가 10곳에                        |
| 3. 흐름 방향 없음   | views가 api를 직접 호출                   |
| 4. 경계 없음        | 모든 속성 public, 외부에서 내부 직접 조작 |
| 5. 비기능 고려 없음 | API 키 노출, 보안 구멍                    |

**1·2번만 잘 잡혀 있어도 스파게티 확률이 급감**해요. 그래서 오늘이 여기에 집중하는 거예요.
